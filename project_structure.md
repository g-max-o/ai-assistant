# Структура проекта

Настоящий документ описывает стандартную организацию каталогов и модулей для Python-проектов,
следуя принципам чистой архитектуры и src-layout.

## Корневая структура
project_root/
├── src/
│ └── package_name/ # Основной пакет приложения
│ ├── domain/ # Бизнес-логика и сущности
│ ├── application/ # Сценарии использования и сервисы приложения
│ ├── infrastructure/ # Реализация взаимодействия с внешним миром
│ └── presentation/ # Входные точки (CLI, API и т.п.)
├── tests/ # Зеркальная структура тестов
├── pyproject.toml
├── .pre-commit-config.yaml
└── README.md

## Слой домена (`domain/`)

Содержит чистую бизнес-логику без зависимостей от фреймворков.

- `models.py` или `entities.py` – доменные сущности (датаклассы, Pydantic-модели с бизнес-поведением)
- `value_objects.py` – неизменяемые объекты-значения (например, `Email`, `Money`)
- `exceptions.py` – иерархия доменных исключений
- `interfaces.py` – интерфейсы (протоколы) репозиториев, сервисов, юнит-оф-ворка
- `events.py` – доменные события (если используется событийная модель)

Правила:
- Никаких импортов из `infrastructure` или `presentation`.
- Разрешены только стандартная библиотека, `pydantic`, `typing_extensions`.

## Слой приложения (`application/`)

Организует сценарии использования (use cases), координирует доменные объекты.

- `services.py` – сервисы приложения, инкапсулирующие бизнес-операции
- `interfaces.py` – порты для внешних зависимостей (например, `EmailSender`, `PaymentGateway`)
- `dto.py` – простые объекты передачи данных между слоями (могут быть датаклассами)
- `commands.py` / `queries.py` – CQRS-команды и запросы (опционально)

Зависимости: только `domain`. Допустим импорт `typing.Protocol` для описания портов.

## Слой инфраструктуры (`infrastructure/`)

Реализует интерфейсы, определённые в `domain` и `application`.

- `database/`
  - `models.py` – ORM-модели (SQLAlchemy)
  - `repositories.py` – реализации репозиториев
  - `unit_of_work.py` – реализация Unit of Work
- `email/` – отправка писем через конкретный SMTP-сервис
- `cache/` – работа с Redis
- `queue/` – постановка задач в очередь (arq, Celery)
- `external_api/` – клиенты к внешним REST/gRPC сервисам (адаптеры)

Каждый модуль обязан реализовывать соответствующий интерфейс из внутренних слоёв.
Инфраструктурный код может зависеть от фреймворков и внешних библиотек.

## Слой представления (`presentation/`)

Обрабатывает ввод/вывод, преобразует запросы в вызовы слоя приложения.

- `api/` – FastAPI роуты, схемы Pydantic для запросов/ответов
- `cli/` – команды Typer/Click
- `middleware/` – промежуточные обработчики (логирование, аутентификация)
- `dependencies.py` – DI-контейнеры (FastAPI Depends)

Зависит от `application` и `infrastructure` (только для внедрения конкретных реализаций).

## Тесты (`tests/`)

Зеркально отображает структуру `src/package_name/`.

- `unit/` – модульные тесты, изолированные с помощью моков на границах
- `integration/` – тесты с реальной БД (testcontainers), кэшем, очередями
- `e2e/` – end-to-end тесты для критических сценариев

Фикстуры размещаются в `conftest.py` на соответствующем уровне.

## Именование модулей

- Избегайте `utils.py`, `helpers.py`. Вместо этого группируйте функции по смыслу.
- Для каждого интерфейса – отдельный модуль с постфиксом `_interface` или `_protocol`.
- Реализации называют по технологии: `postgres_repository.py`, `redis_cache.py`.
