# Правила разработки на Python для qwen3-coder-next

> **Назначение:** данный документ определяет стандарты, подходы и архитектурные решения, обязательные при написании любого Python-кода.  
> **Целевая среда:** Python 3.12+, современный стек библиотек, чистая архитектура.

---

## 1. Общие принципы

- **Читаемость важнее скорости написания.** Код пишется один раз, читается десятки.
- **Явное лучше неявного (PEP 20).** Типы, контракты, конфигурация – всё должно быть выражено явно.
- **DRY, KISS, YAGNI.** Избегай дублирования, излишней сложности и функционала «на вырост».
- **SOLID.** Особое внимание к принципам единственной ответственности и инверсии зависимостей.
- **Композиция предпочтительнее наследования.**
- **Fail fast.** Не скрывай ошибки, обрабатывай их как можно раньше.

---

## 2. Окружение и инструменты

| Категория | Инструмент | Назначение |
|-----------|------------|------------|
| Менеджер пакетов | `uv` или `poetry` | Управление зависимостями и виртуальным окружением |
| Линтер | `ruff` | Проверка стиля и статический анализ |
| Форматтер | `ruff format` | Единообразное форматирование кода |
| Типизация | `mypy` (strict mode) | Статическая проверка типов |
| Тестирование | `pytest` | Фреймворк для тестов |
| Pre-commit | `pre-commit` | Автоматизация проверок перед коммитом |

Все проекты должны включать конфигурационные файлы (`pyproject.toml`, `.pre-commit-config.yaml`) и проходить линтинг и проверку типов без ошибок.

---

## 3. Стиль кода

- **Следовать PEP 8** с максимальной длиной строки **100 символов** (не 79).
- Использовать **`from __future__ import annotations`** во всех модулях для ленивой аннотации типов.
- **Имена:**  
  - Классы – `PascalCase`  
  - Функции и переменные – `snake_case`  
  - Константы – `UPPER_SNAKE_CASE`  
  - Приватные члены – с одним подчёркиванием `_internal`
- **Аннотации типов** – обязательны для всех публичных функций/методов, желательны для приватных.
- **Docstring** в стиле Google или Numpy для всех публичных модулей, классов и функций.
- **Импорты** группировать: стандартная библиотека, сторонние, локальные – с пустой строкой между группами.

---

## 4. Типизация

- Использовать **строгую типизацию** во всём проекте. `mypy` с флагом `--strict`.
- Для передачи данных применять **Pydantic v2** (`BaseModel`). Для простых структур – `TypedDict` или `NamedTuple`.
- Избегать `Any`; если тип неизвестен – `object` или `typing.Protocol`.
- Дженерики: `TypeVar`, `Generic` – когда нужна переиспользуемая логика.
- Для обратных вызовов и зависимостей использовать `Protocol` вместо конкретных классов.
- Применять `TypeGuard` и `isinstance` для сужения типов.

## 5. Структура проекта

Детальное описание: **[project_structure.md](./project_structure.md)**.

Краткие правила:
- **src-layout** (пакет в `src/`).
- Разделение по слоям чистой архитектуры:
  - `domain/` – сущности, value objects, интерфейсы репозиториев
  - `application/` – use cases, сервисы приложения, порты
  - `infrastructure/` – реализации репозиториев, внешние API, БД
  - `presentation/` – CLI, веб-обработчики, схемы API
- Зависимости направлены внутрь: внешние слои зависят от внутренних.

## 6. Асинхронность

- **По умолчанию использовать `async/await`.** Синхронный код допустим только в CPU-bound задачах (с явным обоснованием).
- **Никакого смешивания** синхронных и асинхронных вызовов без необходимости.
- **Рекомендуемые библиотеки с асинхронным API:** `httpx`, `aiofiles`, `asyncpg`, `redis.asyncio`, `aiokafka`, `arq` (фон задач).
- Для запуска асинхронного кода – `asyncio.run()` в точке входа.
- Избегать создания глобального event loop; управление циклами – за фреймворком (FastAPI, etc.).

## 8. Паттерны проектирования

Полный каталог: **[patterns.md](./patterns.md)**.

Обязательные к применению в соответствующих ситуациях:
- **Repository** – абстракция доступа к данным; возвращает доменные объекты.
- **Unit of Work** – управление транзакциями в рамках бизнес-операции.
- **Dependency Injection** – через конструктор или `Depends()` в FastAPI.
- **Factory** – создание сложных объектов, конфигурация зависимостей.
- **Strategy** – подменяемое поведение (например, разные платёжные системы).
- **Decorator** – сквозная функциональность: логирование, метрики, кэширование (использовать `functools.wraps`).
- **Chain of Responsibility** – обработка запросов в цепочке middleware.
- **Observer / Event-driven** – для слабосвязанных компонентов (pyee или простые кастомные сигналы).
- **Adapter** – интеграция с внешними сервисами (всегда оборачивать в свой интерфейс).
- **Facade** – упрощение сложной подсистемы.

При реализации паттернов всегда выделять интерфейс (Protocol или ABC) и его конкретную реализацию.

## 9. Обработка ошибок

- Создавать собственную иерархию исключений в `domain/exceptions.py` (базовый `AppError`, наследники `NotFoundError`, `ValidationError`, `UnauthorizedError` и т.д.).
- Не подавлять исключения без нужды; если подавление необходимо – логировать причину.
- Использовать `@contextlib.contextmanager` или `async with` для ресурсов.
- Все сервисы должны пробрасывать исключения вверх, а не глушить их.
- На уровне контроллеров – перехват исключений и преобразование в HTTP-ответы (FastAPI exception handlers).

## 10. Тестирование

- **Структура:** зеркально повторяет структуру исходного кода (рядом с модулем или в отдельной `tests/`).
- **Типы тестов:**
  - *Unit* – на уровне функций и методов, с моками внешних зависимостей (только на границах).
  - *Integration* – с реальной БД (тест-контейнеры), очередями, файловой системой.
  - *E2E* – минимальное количество, только критически важные сценарии.
- Использовать фикстуры pytest, `pytest-asyncio` для асинхронных тестов, `factory_boy` для генерации объектов.
- **Покрытие кода не самоцель**, но стремиться к 80%+.
- **Не мокать то, чем не владеете** (не мокать внутренности библиотек).
